# 20260402 微信小程序客户端接入方案

- Status: implemented (phase 1 shell)
- Date: 2026-04-02

## 背景与问题定义

当前仓库已经具备一套可运行的浏览器端会议系统：

- PC 浏览器与移动 H5 前端位于 `web/`，技术栈为 `TypeScript + React + Vite`。
- 后端提供认证、会议、审计与 WebSocket 信令接口。
- 媒体链路以浏览器 WebRTC API 为核心，前端 RTC 实现位于 `web/src/rtc.ts`。

现在新增了一个空的小程序标准工程 `wechat/`，目标是在同一仓库中提供微信小程序端能力。

但小程序端不能简单复用当前浏览器端实现，至少存在以下三个结构性差异：

1. 小程序前端运行时不是浏览器环境
   - 现有 `web/` 代码大量依赖 `window`、`fetch`、`WebSocket`、`RTCPeerConnection`、`MediaStream` 等浏览器能力。
   - 小程序端需要改用 `wx.request`、`wx.connectSocket`、小程序页面生命周期和原生组件体系。

2. 当前认证体系默认以服务端会话和 Cookie 为中心
   - 浏览器端通过 `credentials: include` 携带登录态。
   - 小程序端需要先验证微信请求与 Cookie/会话行为在真实设备上的表现，再决定是否继续复用现有会话模式，还是补充显式 token 模式。

3. 小程序音视频能力与浏览器 WebRTC 不是同一套接入面
   - 浏览器端当前实现基于 `RTCPeerConnection` 和 `getUserMedia`。
   - 小程序端需要单独评估可用的实时音视频能力、权限模型、设备限制和与现有信令协议的兼容方式。

因此，本次接入不应把“微信小程序端”视为 `web/` 的简单移植，而应视为一个新的客户端实现。

## 目标与非目标

### 目标

- 在仓库内正式建立独立的微信小程序客户端工程 `wechat/`。
- 明确小程序端与现有浏览器端共用的能力边界：
  - 共用后端认证接口
  - 共用会议查询、加入、离会、结束会议等 REST 接口
  - 共用现有 WebSocket 信令协议，若小程序运行时能力允许
- 以分阶段方式推进，先打通身份和入会主流程，再逐步接入会中能力。
- 不破坏现有 `web/` 前端和生产部署链路。

### 非目标

- 本轮不承诺一次性交付浏览器端全部会中能力到小程序端。
- 本轮不直接复用 `web/src/rtc.ts` 的浏览器实现。
- 本轮不改变现有浏览器前端的设计和路由结构。
- 本轮不把小程序端纳入当前 Docker 生产发布包。
- 本轮不引入新的后端服务或新的数据库。

## 方案概述与核心决策

### 1. 小程序端作为独立客户端工程维护

采用独立工程目录：

- `wechat/miniprogram/`：小程序页面、组件和业务代码
- 保持与 `web/` 并行，而不是混入 `web/`

核心原因：

- 小程序运行时、构建产物、页面模型与浏览器前端差异过大
- 复用仓库，但不强行复用前端工程

### 2. 采用“分阶段交付”而不是一次性覆盖全量能力

建议拆分为三个阶段：

#### 阶段 1：身份与入会壳层

目标：

- 小程序端启动
- 微信快捷登录
- 微信首次登录自动创建账号
- 小程序端显式 token 登录态保持
- 当前用户态展示
- 输入公开 `9` 位会议号并查询会议
- 输入会议密码并加入会议
- 基础会中壳层（标题、成员信息、退出入口）

本阶段不承诺：

- 摄像头、麦克风、共享屏幕
- 白板
- 录制
- 审计统计
- 完整聊天

这是为了先把“能登录、能入会、能到达会中上下文”跑通。

#### 阶段 2：会中基础协作能力

候选能力：

- 成员列表
- 文字聊天
- 主持人结束会议
- 小程序端最小会中状态同步

本阶段仍不承诺浏览器端完整主舞台表现。

#### 阶段 3：音视频能力评估与接入

这一阶段必须在真实设备和微信能力验证通过后再做，不能预先假定浏览器 WebRTC 方案可直接沿用。

需要单独确认：

- 小程序端可用的实时音视频 API
- 与现有信令消息格式的兼容性
- TURN / STUN / 弱网策略
- 前后台切换、来电打断、权限拒绝等小程序端特有场景

### 3. 先复用现有后端认证与会议接口，不急于新开后端协议

第一阶段尽量复用这些现有接口：

- `POST /api/auth/login/code`
- `POST /api/auth/login/verify`
- `POST /api/auth/register/code`
- `POST /api/auth/register/verify`
- `GET /api/auth/me`
- `POST /api/auth/logout`
- `GET /api/meetings/{meetingID}`
- `POST /api/meetings/{meetingID}/join`
- `POST /api/meetings/{meetingID}/participants/{participantID}/leave`

但要明确一个风险点：

- 当前接口的登录态以 Cookie 会话为主
- 小程序端是否能稳定复用这套模式，需要在开发者工具和真机上验证

当前实现已经按该风险点落地了第一阶段最小兼容方案：

- 后端新增 `POST /api/auth/wechat/mini/login`
- 小程序端通过 `wx.login` 获取 `code`
- 后端调用微信 `jscode2session` 接口换取 `openid`
- 后端基于现有 `auth_sessions` 创建会话，并把 `sessionToken` 显式返回给小程序端
- 小程序端后续通过 `Authorization: Bearer <token>` 调用现有认证和会议接口

因此第一阶段的小程序端不依赖 Cookie 维持登录态。

### 4. 小程序端 UI 不直接照搬浏览器端复杂会中布局

小程序端第一阶段采用更适合移动端的简单结构：

- 登录页
- 注册页
- 加入会议页
- 会议密码页
- 基础会中页

优先保证：

- 流程通
- 状态稳定
- 页面可维护

而不是追求与浏览器端 1:1 视觉还原。

## 涉及模块 / 数据结构 / 接口 / 配置 / 存储影响

### 小程序端新增模块

当前第一阶段实际结构：

- `app.ts` / `app.json` / `app.scss`
- `pages/index/`
- `pages/home/`
- `pages/join/`
- `pages/meeting-room/`
- `components/`
- `utils/request.ts`
- `utils/session.ts`
- `utils/api.ts`
- `utils/runtime.ts`

### 现有后端的潜在影响

第一阶段实际已经引入了一个最小后端协议扩展：

- `POST /api/auth/wechat/mini/login`

其余认证与会议接口继续复用现有实现，并通过 Bearer token 访问。

后续若需要在小程序端接入邮箱验证码登录，再进入下一轮扩展，补：

- 小程序端验证码请求与校验页面
- 小程序端验证码限流标识在 Cookie / 显式 client id 下的兼容方案

### 配置与发布影响

- `wechat/` 第一阶段不加入现有 `Makefile` 的 `build/linux/pack/publish` 流程
- 先作为独立客户端工程在仓库内维护
- 生产发布方式在后续小程序首次可用时再单独补文档
- 生产环境需要为后端额外提供：
  - `MEETING_WECHAT_MINIPROGRAM_APP_ID`
  - `MEETING_WECHAT_MINIPROGRAM_APP_SECRET`
  - 可选 `MEETING_WECHAT_MINIPROGRAM_API_BASE_URL`

## 兼容性、迁移方案与风险

### 兼容性

- 不修改现有 `web/` 工程结构
- 不修改现有生产 Docker 部署
- 不修改现有浏览器会议主流程

### 主要风险

1. 小程序端会话保持与验证码限流 Cookie 行为可能与浏览器不同
2. 小程序端实时音视频能力未验证前，不能承诺直接接入现有 Mesh WebRTC 主链路
3. 如果后续改为 token 鉴权，需要同时兼容浏览器端和小程序端两种客户端
4. 小程序审核、域名白名单、WebSocket / request 合法域名都需要额外配置

### 风险控制

- 采用分阶段交付
- 第一阶段先做身份与入会主流程
- 音视频能力单独评估，不预设结果
- 后端协议只在验证必要时才扩展

## 验证与回滚思路

### 第一阶段验证

- 小程序端能启动并进入登录页
- 点击登录页微信图标后能通过 `wx.login` 获取 code 并完成后端登录
- 首次微信快捷登录会自动创建 `wechat_openid` 绑定账号
- 已登录小程序端重开后能通过本地保存的 Bearer session 恢复用户态
- 输入公开 `9` 位会议号能正确查询并加入会议
- 加入失败、密码错误、验证码过频等错误提示正确
- 会中页能显示基础会议标题、会议号、当前身份，并支持离开会议

### 第二阶段验证

- 小程序端能看到会中基础状态
- 能退出会议
- 聊天或成员列表等基础协作能力正常

### 第三阶段验证

- 真机摄像头/麦克风权限申请
- 建链成功率
- 弱网和前后台切换
- 与浏览器端互通

### 回滚思路

- 小程序端作为独立客户端目录存在
- 若任一阶段失败，可停止发布小程序版本，不影响现有浏览器端
- 若后端为小程序增加了新鉴权接口，应保持与旧浏览器逻辑兼容，避免回滚困难

## 备选方案及放弃原因

### 方案 A：直接把 `web/` 改造成多端统一工程

放弃原因：

- 当前浏览器端大量使用浏览器专属 API
- 小程序端运行时差异过大
- 会显著提高现有前端复杂度并增加回归风险

### 方案 B：一次性实现小程序端完整视频会议能力

放弃原因：

- 技术风险高
- 范围过大
- 无法在缺少真实设备验证前作出可靠承诺

### 方案 C：小程序端只做壳，不接现有后端

放弃原因：

- 与当前仓库目标不符
- 会产生重复认证与重复会议模型

## 决策结论

- 在仓库内保留独立的 `wechat/` 小程序客户端工程。
- 先按“身份与入会壳层 -> 基础会中协作 -> 音视频能力评估”三阶段推进。
- 第一阶段优先复用现有认证与会议接口，并已新增小程序专用的微信快捷登录入口：
  - `POST /api/auth/wechat/mini/login`
- 小程序端登录态采用显式 `sessionToken` + `Authorization: Bearer`，不依赖 Cookie。
- 浏览器端与小程序端继续共用同一套 `auth_sessions` 存储，不新增独立认证服务。
