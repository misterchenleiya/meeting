# 验证码发送限流加固

- 状态：implemented
- 日期：2026-04-02

## 背景与问题定义

当前验证码发送仅按“邮箱 + 用途”限制 60 秒重发。同一个用户只要修改邮箱地址，就可以立即再次请求验证码，无法满足“每次发送验证码至少间隔 1 分钟”的防刷要求。

同时，如果直接按 IP 做 60 秒硬限制，会误伤同一局域网或同一 NAT 出口下的其他正常用户。

## 目标与非目标

### 目标

- 保留同一邮箱 60 秒重发限制。
- 新增匿名客户端级别的 60 秒全局冷却，防止用户通过修改邮箱绕过限制。
- 新增 IP 维度的宽松兜底限流，降低恶意批量请求风险。
- 刷新页面后仍然生效，不依赖前端内存状态。

### 非目标

- 不引入图形验证码或第三方风控。
- 不阻断同一局域网内的正常少量并发使用。
- 不改变注册、自动注册、密码登录的业务语义。

## 方案概述与核心决策

采用三层限流：

1. 邮箱 + 用途：60 秒
2. 匿名验证码客户端：60 秒，全局生效，不区分邮箱
3. IP 兜底：10 分钟内最多 12 次验证码发送

匿名验证码客户端由服务端通过 `HttpOnly` cookie 下发。这样用户刷新页面也无法绕过限流；但如果清理浏览器状态或切换到全新客户端，仍由 IP 维度兜底。

## 涉及模块 / 数据结构 / 接口 / 配置影响

- `internal/auth`
  - 增加验证码请求元信息（客户端 ID / IP）
  - 在发码前执行三层限流检查
- `internal/httpapi`
  - 为验证码请求自动分配匿名客户端 cookie
  - 将限流错误映射为中文 `429` 提示
- `internal/storage/sqlite`
  - 在 `auth_verification_codes` 中增加 `client_id`、`ip_address`
  - 基于现有验证码记录做客户端/IP 查询，不额外引入独立限流表

## 兼容性、迁移方案与风险

- 旧库通过 schema 增量补列，兼容已有数据。
- 旧客户端第一次请求验证码时会自动收到新的匿名客户端 cookie。
- 同一 NAT 出口下的多个用户仍共享 IP 兜底配额，因此 IP 限制必须保持宽松。

## 验证与回滚思路

### 验证

- 同一邮箱 60 秒内不能重复发码
- 修改邮箱后，同一浏览器 60 秒内仍不能再次发码
- 刷新页面后 60 秒限制仍然存在
- 同一 IP 在 10 分钟内达到 12 次后触发兜底限流

### 回滚

- 若误伤较多，可先放宽 IP 窗口或次数
- 如需完全回退，可移除客户端/IP 限流，仅保留邮箱维度限制
