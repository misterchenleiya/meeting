# Meeting Open Issues

- Status: open
- Updated: 2026-03-31
- Scope: 基于项目最初需求与当前仓库实现情况的差距分析

## 说明

本文档不是架构方案，而是当前阶段的待办与遗留问题清单，便于后续拆分为真正的 Issue、里程碑或任务卡片。

优先级说明：

- `P0`：阻塞核心需求验收，必须优先完成
- `P1`：重要功能缺口，建议尽快补齐
- `P2`：增强项或工程化收尾，可在主链路稳定后推进

## 当前已基本满足的需求

以下能力已经具备可运行雏形，不在本文档中重复作为主缺口列出：

- 会议创建、加入、离开、主持人结束会议
- 主持人 / 助理 / 参会者基础角色模型
- `participant` 默认仅聊天，其余能力需主持人授权
- 1v1 `WebRTC` P2P 主链路
- 本地媒体采集、本地/远端视频预览
- ICE / STUN / TURN 配置化与基础断线重连
- 屏幕共享
- 本地录屏/录音缓存、下载、丢弃
- 白板协作
- 文字聊天
- 就位确认
- 临时会议纪要运行态
- 临时纪要本地导出
- 基础审计统计上报
- 昵称修改并写入聊天留痕
- 会议结束后删除运行态数据

## 遗留问题

| ID | Severity | 问题 | 影响 | 建议处理 |
| --- | --- | --- | --- | --- |
| TECH-001 | P0 | 会议运行态完全保存在单进程内存 | 服务重启后房间、白板、聊天、就位确认都会丢失 | 增加断线重连策略，必要时补轻量级临时态快照机制 |
| TECH-002 | P0 | WebSocket `CheckOrigin` 当前开发期全放开 | 生产环境存在跨站连接风险 | 改为可配置白名单并区分开发/生产模式 |
| TECH-003 | P0 | 密码处理仍是简单 `SHA-256 + salt` | 不适合作为正式注册登录密码方案 | 登录体系落地时改为 `argon2id` 或 `bcrypt` |
| TECH-004 | P0 | 缺少真实认证、会话和接口访问控制 | 创建会议、修改昵称、上报审计等操作没有完整身份边界 | 引入真实身份认证、中间件和权限校验 |
| TECH-005 | P1 | STUN 服务器写死为公共 Google STUN，未配置化 | 已支持通过环境变量或 JSON 运行时配置下发 ICE / TURN 服务器 | 仍需要在生产部署侧真正提供 coturn / 私有 STUN 配置 | 将 ICE/TURN 配置接入部署和环境管理 |
| TECH-006 | P1 | 前端录制只适合当前已接入流，动态混排仍较弱 | 参会者中途加入或切流时，本地录制体验不完整 | 为录制层增加动态流编排和布局刷新 |
| TECH-007 | P1 | 白板能力目前仅支持自由笔迹和清空 | 与真实会议白板相比功能偏弱 | 增加文本、形状、撤销/重做、橡皮擦 |
| TECH-008 | P1 | 缺少前端自动化测试和端到端测试 | 媒体协商、权限事件、UI 回归难以及时发现 | 增加前端单测和双浏览器 E2E 场景 |
| TECH-009 | P1 | 断开 `WSS` 后只标记离线，不具备完整的断线恢复流程 | 已补基础的指数退避自动重连与重建信令流程 | 复杂网络抖动下仍需要更细的恢复覆盖和会话回收验证 | 继续补全断线恢复、超时回收和重新协商逻辑 |
| TECH-010 | P1 | 临时纪要目前主要依赖事件文本追加 | 纪要结构化程度不高，后续导出和检索能力受限 | 引入结构化纪要模型，再输出为文本 |
| TECH-011 | P2 | 当前前端页面仍是综合控制台形态 | 功能齐全但不够接近正式产品 UI | 后续拆分页面结构、优化交互流和移动端适配 |
| TECH-012 | P2 | 构建产物收敛到 `build/` 后，历史本地残留如 `web/dist` 仍可能存在 | 容易干扰本地排查和目录认知 | 清理旧产物并统一使用 `make build` |

## 建议推进顺序

本轮实际执行顺序按 `1 -> 2 -> 4` 推进，避免先补外围体验再返工主链路：

- `1. P0 媒体主链路`
  `TURN/coturn`、多 peer Mesh、带宽自适应、断线恢复
- `2. P0 身份体系`
  主持人注册准入、邮箱验证码、密码登录、微信登录
- `4. P1 工程化与安全`
  Origin 白名单、密码哈希升级、前端 / E2E 测试
- 其余项（邀请链接、二维码、会议结束保存纪要提示、白板增强、正式 UI、移动端深入适配）后置，等主链路稳定后再补齐。

以下条目直接来自 `docs/proposals/20260331-media-reliability-and-identity-plan.md`，用于进入实现阶段时拆分任务和排期。

### Phase 1 媒体可靠性

| ID | Priority | 任务 | 涉及模块 | 完成标准 |
| --- | --- | --- | --- | --- |
| MR-001 | P0 | ICE servers 配置化 | `web/src/rtc.ts`、`web/src/App.tsx`、环境变量配置 | STUN / TURN 地址可以通过配置下发，前端不再硬编码单一公共 STUN |
| MR-002 | P0 | 接入 coturn 和 TURN fallback | `web/src/rtc.ts`、部署配置 | 直连失败或 UDP 不可达时，可以退化到 TURN relay，不再只有“能连/不能连”两种状态 |
| MR-003 | P0 | 连接质量监测与自适应治理 | `web/src/rtc.ts`、`web/src/App.tsx` | 利用 `getStats()` 结果调节码率 / 分辨率 / 优先级，并能输出稳定的连接质量摘要 |
| MR-004 | P0 | 多人 Mesh 协商与稳定性加固 | `web/src/rtc.ts`、`internal/signaling/hub.go` | 2 人、4 人、6 人、10 人的多 peer 场景可以稳定协商和重连 |
| MR-005 | P1 | 断线恢复与重新协商流程 | `web/src/rtc.ts`、`web/src/signaling.ts`、`web/src/App.tsx` | ICE / WebSocket 断开后可以自动恢复，不需要手动刷新页面 |
