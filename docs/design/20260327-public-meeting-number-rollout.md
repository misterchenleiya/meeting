# Public 9-Digit Meeting Number Rollout

- Status: implemented
- Updated: 2026-03-27
- Scope: 为会议系统新增公开的 `9` 位数字会议号，用于创建后展示、分享、扫码、加入会议和会中信息展示；内部运行态继续保留当前 opaque room id，避免一次性改动 WSS 与运行时状态机。

## 背景与问题定义

当前系统中的会议标识是后端生成的十六进制字符串，既用于后端内存索引和信令路由，也直接暴露在前端分享窗口、二维码和加入会议输入框中。

这个做法有几个问题：

1. 对用户来说会议号过长，不利于口头传达、手工输入和截图识别。
2. 当前分享窗口虽然已经按设计稿收敛了样式，但核心标识仍然不是用户可直接理解的“会议号”。
3. 会议标识兼任“内部 room key”和“用户可见会议号”两个职责，导致后续任何 UI 调整都会触碰底层运行链路。

因此，本轮需要引入一个真正面向用户的 `9` 位数字会议号，并把会议创建、分享、扫码和加入流程统一到这个公开会议号之上。

## 目标

1. 为每个会议生成一个唯一的 `9` 位数字会议号，作为用户唯一可见的会议标识。
2. 前端所有用户可见位置统一展示这个会议号，并按 `3-3-3` 形式分组显示，例如 `821 503 974`。
3. 加入会议、二维码识别、邀请分享都支持使用这个 `9` 位会议号。
4. 尽量不破坏现有会中运行态、WSS 建链、参与者状态恢复和服务端审计逻辑。

## 非目标

1. 不在本轮重写 WebSocket 房间管理模型。
2. 不在本轮移除内部已有的 opaque room id。
3. 不在本轮引入数据库持久化会议主表，会议仍然保留当前内存生命周期。
4. 不在本轮改变 `joinCode`、权限、录制、聊天、白板等业务规则。

## 方案概述与核心决策

### 1. 引入“公开会议号”和“内部房间 ID”双标识

- `Meeting.ID` 继续保留为内部 room id，用于：
  - 会中运行态索引
  - 现有 WSS 连接
  - 后端审计事件中的会议标识
- 新增 `Meeting.MeetingNumber`，作为公开 `9` 位数字会议号，用于：
  - 创建会议后的展示
  - 分享窗口
  - 复制会议号
  - 邀请二维码
  - 加入会议输入
  - 扫码识别后的会议号回填

这样做的原因是：

1. 改动面更可控，不需要一次性替换所有运行时依赖。
2. 会中恢复和信令回调仍可沿用当前内部 ID，降低回归风险。
3. 用户可见层和内部路由层职责分离，后续再调整会议号形态时不会再次牵动底层链路。

### 2. 后端为公开会议号建立独立映射

- `meeting.Service` 除现有 `meetings map[string]*Meeting` 外，新增公开会议号到内部 ID 的映射。
- 创建会议时生成唯一 `9` 位数字会议号，并登记映射。
- 会议结束或最后一位参会者离开后，删除对应映射。

### 3. 服务端读取会议时统一支持“内部 ID / 公开会议号”双解析

后端所有按 `meetingID` 取会议信息的入口，在服务层统一做解析：

- 若传入值能直接命中内部 `meetings` map，则按内部 ID 处理。
- 否则尝试按公开会议号映射到内部 ID。

这样可以兼容两类调用：

1. 前端加入会议页输入 `9` 位会议号时，通过 `GET /api/meetings/{meetingID}` 正常查到会议。
2. 会中运行态继续使用内部 ID 调用现有接口和 WSS，而无需同步替换整条链路。

### 4. 前端展示层统一使用公开会议号

前端 `Meeting` 类型新增 `meetingNumber` 字段。

用户可见位置统一改为显示 `meetingNumber`，并通过统一格式化函数按 `3-3-3` 展示：

- 加入会议输入确认提示
- 分享窗口标题与二维码说明
- 复制会议号
- 分享文案中的会议号
- 扫码回填后的提示文案

### 5. 分享和二维码改为公开会议号语义

- 分享二维码和分享文案中的 URL 不再携带内部 room id，而是携带公开会议号。
- 二维码解析时同时兼容旧字段：
  - 优先读取 `meetingNumber`
  - 若没有，再回退读取旧的 `meetingId`
- 文本二维码场景下，如果内容本身就是一串数字，则按会议号处理。

### 6. 加入输入对空格和分组展示做兼容

由于 UI 显示是 `3-3-3` 分组形式，用户可能复制或手输带空格的会议号。

前端与后端统一按“规范化会议号”处理：

- 输入时允许 `821 503 974`
- 请求前转成 `821503974`
- 服务端解析公开会议号时也做去空格处理

## 涉及模块 / 数据结构 / 接口 / 配置 / 存储影响

### 后端

- `internal/meeting/types.go`
  - 为 `Meeting` 新增 `MeetingNumber string`
- `internal/meeting/service.go`
  - 生成唯一 `9` 位会议号
  - 维护公开会议号映射
  - 服务层统一支持内部 ID / 公开会议号双解析
- `internal/httpapi/server.go`
  - 无需改路由形态，但会自然支持公开会议号查找
- `internal/signaling/hub.go`
  - 继续沿用内部 room id，无需改协议结构

### 前端

- `web/src/types.ts`
  - `Meeting` 新增 `meetingNumber`
- `web/src/App.tsx`
  - 分享窗口、复制会议号、二维码、加入会议、扫码回填改为使用公开会议号
  - 新增会议号格式化与规范化 helper
- `web/src/api.ts`
  - API 入参名称可以继续保留 `meetingId`，但值允许传公开会议号

## 兼容性、迁移方案与风险

### 兼容性

1. 现有会中会话、WSS 路径和内部审计仍使用内部 ID，不会因会议号改造整体失效。
2. 分享二维码解析兼容旧 `meetingId` 字段，避免旧二维码在本地联调时立即失效。
3. 服务端统一解析后，旧的内部 ID 路径在实现期内仍然可用。

### 迁移方案

1. 先引入 `meetingNumber` 和映射层。
2. 再把前端所有用户可见显示切换到 `meetingNumber`。
3. 最后补测试，覆盖：
  - 创建会议生成 `9` 位会议号
  - 通过会议号查会
  - 通过会议号加入
  - 旧内部 ID 路径继续兼容

### 风险

1. 如果会议号唯一性处理不严谨，可能出现冲突或覆盖错误。
2. 如果有代码仍在 UI 层直接使用 `meeting.id`，会出现部分页面还是显示旧内部 ID。
3. 如果分享二维码与加入页的解析规则不一致，可能出现“复制的是会议号，加入时却查不到”的问题。

## 验证与回滚思路

### 验证

1. 创建会议后，前端分享窗口显示 `9` 位会议号。
2. 分享窗口复制的会议号为公开会议号，而不是内部十六进制 ID。
3. 通过加入页输入 `9` 位会议号可以正常查会并加入。
4. 扫码识别分享二维码后，加入页能正确回填公开会议号。
5. 会中信令连接、刷新恢复、结束会议和审计日志不回归。

### 回滚

若上线后发现会议号解析有严重问题，可回滚到仅显示内部 ID 的版本，并保留本次新增的设计文档作为下一轮重做依据。
