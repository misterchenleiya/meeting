# Meeting Product UI Rollout

- Status: implemented
- Updated: 2026-03-25
- Scope: 根据 `docs/design/meeting-auth-preview.html` 与 `docs/design/meeting-room-preview.html` 落地当前前端主界面

## 背景与问题定义

当前前端已经具备会议创建、加入、WebRTC、白板、聊天、录制、审计等能力，但主页面仍是“综合控制台”形态，不接近正式产品入口和会中舞台体验。

本轮需要把现有能力迁移到更接近产品化的黑色风格 UI：

1. 入会前页面保持当前黑色风格与左右分区布局语义不变，但改为铺满浏览器窗口，不再保留外层留白和卡片式边界。
2. 增加登录、登录后入口、预定会议、快速会议、加入会议和密码弹窗。
3. 会中页改为黑色主舞台 + 右侧缩略窗 + 底部工具栏。
4. 成员与聊天默认隐藏，改为右侧抽屉。

## 目标

1. 让前端主流程与最新 `docs/design` 预览保持一致。
2. 不破坏当前已实现的创建 / 加入 / WebRTC / 白板 / 聊天 / 录制 / 审计主链路。
3. 在后端能力未补齐前，通过兼容实现先完成产品化 UI 过渡。

## 非目标

1. 不在本轮实现真实邮箱登录、验证码、会话系统。
2. 不在本轮实现真正的预定会议持久化。
3. 不在本轮实现基于 `joinCode` 的新加入接口。
4. 不在本轮实现新的后端媒体状态同步协议。

## 方案概述与核心决策

### 1. 登录流

- 登录页按设计稿落地为本地前端状态，不接真实认证接口。
- 登录成功后仅在当前前端状态中保留邮箱信息，用于生成主持人默认 `userId` 和昵称。
- 登录预览默认铺满浏览器窗口，不再保留外层预览控制条的可见占位。
- 左侧仅保留 `meeting` 品牌标识，并通过聚光灯效果维持视觉焦点。
- 右侧登录框改为居中悬浮卡片，在登录视图下以全屏浮层方式渲染，并取消右侧独立分栏的空白区域；卡片仅保留邮箱和密码输入框、获取临时验证码按钮、忘记密码入口、登录按钮、圆形居中的微信登录图标和加入会议入口。
- 登录态内的卡片切换统一只发生在同一页壳层中，登录卡片、登录后入口卡片、预定会议卡片和加入会议卡片都保持同一套页面背景与卡片槽位，不做整页跳转。
- 登录态到登录后入口的切换只更新页面内的卡片状态，不改写 URL hash，避免视觉上出现跳页感。
- 顶部增加仅供预览使用的状态切换控制栏，用于在登录、登录后入口、预定会议、加入会议和密码弹窗之间切换。
- 登录视图的三段主元素按从上到下排列：`meeting`、聚光灯效果、登录卡片；登录卡片不再垂直居中，而是向下下移到聚光灯下方。
- 登录视图的三段主元素改为整体居中堆叠显示，并统一全页背景，避免登录卡片把页面切成上下两块不同色调的区域。
- 登录页的聚光灯背景改为由全屏根容器承载，品牌区不再局部裁切背景效果，保证光晕铺满整个屏幕。
- 登录页的聚光灯进一步收束为 `meeting` 文字下方的局部舞台光，不再向卡片区域大面积扩散。
- 邮箱和密码的静态提示文字移除，改为输入框占位提示 `请输入邮箱`、`请输入密码`。
- 登录卡片在邮箱输入框上方补充 `邮箱验证码登录` 文案，并提供 `使用密码登录 >` 的蓝色切换入口。

### 2. 预定会议与快速会议

- 快速会议直接复用现有 `POST /api/meetings`。
- 预定会议 UI 完全按设计稿展示，但因当前后端无预约会议模型，本轮仍复用 `POST /api/meetings` 立即创建会议。
- 时区和会议时间先作为前端侧输入保留，用于后续扩展，不进入当前后端接口契约。

### 3. 加入会议与密码弹窗

- 加入会议采用两段式交互：
  1. 先输入会议号和昵称；
  2. 调用现有 `GET /api/meetings/{meetingID}` 做预检；
  3. 预检成功后弹出密码悬浮窗，再调用现有 `POST /api/meetings/{meetingID}/join`。
- 由于当前后端仍以 `meetingID` 作为加入路径，本轮前端中的“会议号”实际映射到 `meetingID`，不强行切到 `joinCode`。
- 加入会议卡片只保留会议号与昵称输入，不再展示会议主题和状态预检摘要，减少冗余信息。

### 4. 会中主舞台

- 会中主舞台按设计稿改为两种状态：
  - 无视频 / 无共享：中央头像墙
  - 有视频 / 共享：主画面 + 右侧缩略窗
- 默认主画面优先选择主持人的活动画面；若主持人无活动画面，则降级到首个可用画面。
- 右侧缩略窗支持双击切换主画面。

### 5. 媒体状态的兼容实现

- 当前后端未同步完整“谁正在开摄像头 / 谁正在共享屏幕”的结构化状态。
- 因此本轮前端按以下规则近似判断：
  - 本地：根据本地媒体流与本地共享状态判断；
  - 远端：根据当前远端媒体流是否存在视频轨道判断是否进入主舞台候选。
- 昵称旁的麦克风状态优先使用本地实际轨道状态，对远端则回退到当前参与者快照中的媒体偏好字段。

### 6. 已实现功能的保留策略

- 白板、权限申请、就位确认、临时纪要、审计、事件流等能力仍保留。
- 这些内容不再占据会中主舞台，而是收敛为下方辅助面板，避免破坏设计稿的主视觉结构。

## 兼容性、风险与迁移

### 兼容性

- 现有后端接口保持不变。
- 现有 `WSS` 信令协议保持不变。

### 风险

1. “预定会议”因后端能力限制，会表现为“按预定表单创建并立即进入会议”。
2. “会议号”本轮仍对应 `meetingID`，与未来可能切换到 `joinCode` 的产品语义存在偏差。
3. 远端摄像头 / 共享屏幕状态判断仍是前端近似推断，不是服务端权威状态。

### 迁移方向

- 后续如补齐登录体系、预约会议模型和按 `joinCode` 加入接口，前端当前页面结构可以直接复用，只替换接口实现。

## 验证与回滚思路

### 验证

1. `npm run build`
2. `make run-frontend`
3. 手动验证登录、快速会议、预定会议、加入会议、密码弹窗、邀请弹窗、成员 / 聊天抽屉与主舞台切换。

### 当前结果

- 当前前端已按本方案完成主界面切换。
- 登录页、登录后入口、预定会议、加入会议、密码悬浮窗、会中主舞台与右侧抽屉均已接入现有前端主流程。
- `npm run build` 已通过。

### 回滚

- 若本轮产品化 UI 出现阻塞问题，可回滚 `web/src/App.tsx`、`web/src/api.ts` 和 `web/src/styles.css` 到旧版综合控制台。
