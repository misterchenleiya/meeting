# Meeting Room Screenshot-Style Single Page Layout Proposal

- Status: implemented
- Updated: 2026-03-27
- Scope: 将会议房间改为参考截图的深色单屏会议室布局，主画面全屏，底部工具栏尽量贴近参考样式，参会者头像采用无边框圆形样式，并保留主持人 / 参会者双菜单作为隐藏层

## 背景与问题定义

当前会议房间页面虽然已经具备主舞台、成员抽屉、聊天抽屉、白板、录制、纪要、审计等能力，但整体仍然是“主舞台 + 侧边栏 + 下方辅助面板”的复合布局。

这个结构的问题是：

1. 视觉焦点会被多个区域分散，用户很难把注意力稳定放在会议主画面上。
2. 工具数量继续增加时，页面会越来越拥挤。
3. 常用操作与管理操作混在同一层级，不利于参会者快速理解自己的可见能力。
4. 在移动端和窄窗口环境下，现有布局的空间压力会更明显。

因此，本次希望把会议房间重构为参考截图那种单一页面的深色会议房间：

- 主画面始终占满当前浏览器窗口
- 参会者头像只保留圆形头像、麦克风状态和昵称
- 昵称在 hover 时才出现编辑按钮，点击后弹出临时修改窗口
- 底部工具栏尽量贴近截图中的深色 dock 风格
- 主持人工具继续仅对主持人和助理可见，但默认不占据主视觉

## 目标

1. 让会议主画面始终成为唯一的持续视觉焦点。
2. 将所有辅助能力收纳到菜单或覆盖层中，避免布局被多个固定面板切碎。
3. 清晰区分参会者自助操作与主持方管理操作。
4. 保留现有会议能力，不在本轮缩减功能，只改变信息架构和页面承载方式。
5. 参考截图优化头像、昵称、工具栏和弹窗的视觉重量，减少明显卡片边框和模块割裂感。

## 非目标

1. 不在本轮修改会议创建、加入、身份认证、WebRTC、信令或存储模型。
2. 不在本轮引入新的媒体协议或新的后端角色模型。
3. 不在本轮更改当前权限规则，只调整这些能力在界面中的组织方式。
4. 不在本轮重做登录页和加入页布局。

## 方案概述与核心决策

### 1. 页面结构改为“单一主舞台 + 菜单覆盖层”

- 会议房间页面以全屏主舞台作为唯一底层内容。
- 主舞台始终占满当前浏览器窗口，不再保留固定的右侧栏或下方功能区。
- 所有辅助内容以覆盖层、弹层、抽屉、底部面板或菜单的方式浮在主舞台之上。
- 若当前没有视频或屏幕共享，主舞台仍保持全屏，只是在中间显示头像墙、占位画面或状态提示。
- 头像墙优先采用无边框圆形头像、麦克风状态和昵称的组合，不再使用明显卡片边框去切分视觉焦点。

### 2. 工具收敛为两个菜单

#### 参会人员工具菜单

该菜单对所有参会者可见，承载参会者的自助操作，建议包含：

- 麦克风开关
- 摄像头开关
- 共享屏幕
- 聊天
- 成员列表
- 邀请参会者
- 修改昵称
- 申请权限
- 离开会议

如果当前能力未被授予，则对应菜单项保持可见但呈现禁用、申请或说明态，而不是把功能完全隐藏。

#### 主持人工具菜单

该菜单仅对主持人和助理可见，承载会议管理操作，建议包含：

- 授权参会者能力
- 指定助理
- 发起 / 处理就位确认
- 录制控制
- 临时纪要导出 / 保存提醒
- 白板清空或管理操作
- 审计与事件查看
- 结束会议

主持人菜单的可见性由角色决定，不由当前是否已授予某项权限决定。也就是说，菜单先按角色分层，再按具体权限决定每个动作是否可执行。

这两类菜单继续保留，但在视觉上不再作为主画面上的固定栏位，而是收纳到隐藏层、下拉层或临时弹层中。

### 3. 主画面保持全屏，其他内容变为“按需出现”

主舞台始终是页面底层：

- 有活动视频或屏幕共享时，活动内容占据舞台核心区域
- 没有活动媒体时，显示头像墙或空状态提示
- 连接状态、会议标题、角色信息等只保留为轻量级角标或浮层
- 成员、聊天、白板、权限、录制、纪要等都不再占固定版面，而是通过菜单打开

### 4. 头像与工具栏视觉细节

- 参会者头像没有明显边框，只显示圆形头像。
- 主持人默认昵称为 `主持人`，其他未设置或未修改昵称的参会者默认显示为 `匿名用户`。
- 头像下方只保留麦克风状态和昵称，二者在同一行内居中排列，麦克风图标直接复用工具栏同款图标并缩小显示，不再出现偏左或同一行错位。
- 鼠标移动到昵称上时显示编辑按钮，按钮悬浮在昵称右侧，不参与昵称本身的居中排布；编辑入口只保留紧凑图标按钮，不再显示额外说明文字，点击后弹出临时窗口。
- 修改窗口的输入框默认加载当前昵称，便于直接改名。
- 顶部左侧仅保留会议信息和品牌标识，不再保留窗口控制按钮或额外状态按钮。
- 顶部左侧按 `会议详情 → 网络状态 → 分享` 的顺序组织信息和入口，再加上主持人 / 会议两个工具菜单，形成轻量但完整的顶栏操作区。
- 会议详情下方继续只保留当前会议持续时间；会议 ID 不在标题栏单独显示，而是收纳到分享窗口内部，使用 `9` 位数字并按 `3-3-3` 形式分组显示，例如 `821 503 974`，复制会议 ID 的图标按钮需要紧跟在会议号这一行的末尾显示，不与右上角关闭按钮混淆。
- 网络状态入口不再点击打开居中弹窗，改为鼠标 hover 后在按钮正下方显示固定位置的信息浮层；光标移出后浮层立即消失。浮层只展示四项摘要：`网络状态`、`延迟`、`丢包率`、`传输速率`，其中传输速率使用 `MB/s` 并保留两位小数。
- 顶部右侧在 `会议工具` 后补 `设置` 和 `进入全屏` 两个轻量按钮，其中 `设置` 提供会议设置预览入口，`进入全屏` 保留为接近截图的图标化入口。
- `进入全屏` 按钮在 hover 时显示 `进入全屏模式` 提示，进入全屏后切换为 `退出全屏模式` 提示，并同步切换图标样式。
- 底部工具栏采用接近参考截图的深色 dock 风格，按钮以图标加标签为主，去掉核心按钮下方多余符号和元宝纪要入口，不再保留左侧聊天输入和举手入口，仅保留中部工具组与右侧结束会议按钮；顶部菜单、底部按钮和部分菜单项点击后会切换预览态，展示对应的示意面板或状态变化。
- 聊天入口改为右侧浮动聊天窗口：位置继续停留在会议画面右侧，但显示后不再压缩或重排主舞台，只作为覆盖在会议上的浮窗出现。
- 工具栏聊天按钮需要支持未读消息提示：只在按钮右上角显示一个红点，不显示数量；打开聊天浮窗后红点消失。
- 当任一参会者与后端的 WSS 连接中断或异常时，整个会议进入灰色模式，并在屏幕中央显示一个不可关闭、不可拖动的网络异常浮动窗口；连接恢复后该窗口自动消失。
- 分享窗口采用连续信息排版而不是独立信息组：顶部显示 `meeting` 与会议号，中部左侧展示连续的会议信息和加入链接，右侧展示会议二维码，二维码下方显示 `会议ID：821 503 974`；底部操作按钮保留 `复制全部信息` 和 `复制会议二维码`。
- 贴附式子窗口统一分为两类：`底部贴合型` 和 `顶部贴合型`。二者都优先沿触发按钮对齐，只有在即将超出视口边界时才做水平钳制，不再回退到屏幕居中。
- `底部贴合型` 适用于底部工具栏按钮，例如 `应用`、`结束会议`。子窗口默认出现在触发按钮上方，底边与工具栏顶部之间保持统一的小间距，视觉上像从 dock 上沿弹出。
- `顶部贴合型` 适用于顶部标题栏按钮，例如 `主持人工具`、`会议工具`、`设置`。子窗口默认出现在触发按钮下方，顶边与标题栏底边之间保持统一的小间距，视觉上像从 topbar 下沿下拉。
- 结束会议按钮采用截图中的“按钮式菜单”形态：点击后在右下角结束会议按钮附近弹出小型操作卡片，按钮本身切换为 `取消`，卡片内只保留 `全员结束会议` 和 `离开会议` 两项操作，不再提供 `AI托管` 入口。
- 结束会议按钮的操作卡片需要锚定在底部右侧结束会议按钮上方，并根据底部工具栏高度单独计算；卡片采用固定定位避免被外层裁切，底边与工具栏顶部之间保留明确安全间距，确保整张卡片完整可见且不被底部工具栏遮挡。

### 5. 角色可见性保持明确

- `participant`：只看到参会人员工具菜单。
- `host`：同时看到参会人员工具菜单和主持人工具菜单。
- `assistant`：同时看到参会人员工具菜单和主持人工具菜单，但具体危险操作仍应继续遵循当前权限规则。

这次改版的核心是“菜单分层”和“视觉收束”，不是“权限重写”。

### 6. 覆盖层优先，避免页面跳布局

为了保持主画面稳定，所有辅助界面统一采用覆盖层策略：

- 成员列表：浮层或抽屉
- 聊天：默认使用右侧浮动窗口，不压缩主舞台宽度
- 邀请二维码：模态框
- 录制权限申请：模态框
- 就位确认：居中弹窗
- 录制与纪要：菜单内页或底部面板
- 白板：独立覆盖层，必要时切换为沉浸式编辑态
- 所有弹窗右上角统一提供接近 macOS 风格的小红色圆形关闭按钮，和底部取消/返回按钮配合使用，降低关闭成本。
- 网络状态改为 hover 浮层后，不再进入统一的子窗口开关体系，也不再提供关闭按钮或 `ESC` 关闭逻辑。
- 网络异常浮层独立于普通子窗口体系，不提供关闭或移动能力，只由网络状态恢复自动清除。
- 所有子窗口统一遵循同一关闭规则：右上角关闭按钮、重复点击同一触发按钮、`ESC` 都可以关闭；再次点击其他会打开子窗口的按钮时，会先关闭当前子窗口再打开目标子窗口；子窗口外的空白区域点击无效。
- 除网络异常浮层外，其他覆盖层打开时底层主舞台继续保持原有彩色视觉；聊天浮窗和其他覆盖层都不再压缩成窄视图。
- 应用、结束会议、主持人工具、会议工具和设置这几类子窗口统一采用贴附式定位；其余子窗口继续使用屏幕居中显示，保持主舞台视觉稳定。
- 设计预览页顶部的“预览控制”工具条默认显示，便于直接切换舞台、菜单、网络状态和聊天未读等预览态。

## 兼容性、迁移方案与风险

### 兼容性

- 会议创建、加入、权限、聊天、白板、录制、审计等后端接口可以暂时保持不变。
- 现有主舞台数据结构可以继续复用，只是展示层从分栏布局改为全屏布局。

### 迁移方案

建议分两步推进：

1. 先把房间页面外壳改成全屏单页结构，保留现有能力入口。
2. 再把当前右侧抽屉和下方功能面板逐步迁移进两个菜单。

这样可以把“结构变化”和“功能迁移”拆开，降低一次性改动风险。

### 风险

1. 工具入口收敛后，部分功能的发现成本会变高，需要清晰的图标、标签和菜单分组。
2. 主持人菜单和参会人员菜单的边界如果定义不清，容易让助理和普通参会者产生困惑。
3. 如果菜单层级过深，常用操作会比当前侧栏模式更慢。
4. 白板、录制、纪要等能力如果全部塞进同一菜单，信息密度会过高，需要后续再拆分分组。

## 验证与回滚思路

### 验证

1. 会议房间在桌面端始终保持全屏单页，不出现固定侧栏占位。
2. 无论是否有视频流，主舞台都不离开当前视口。
3. 参会者头像不再出现明显卡片边框，昵称和麦克风状态布局保持简洁。
4. 鼠标移到昵称上时，编辑按钮才出现，点击后默认带出当前昵称。
5. 参会人员菜单仅显示参会者可见项。
6. 主持人菜单仅对主持人和助理可见。
7. 成员、聊天、邀请、录制、就位确认等覆盖层打开后不会改变主舞台布局。
8. 在窄窗口与移动端尺寸下，菜单仍可正常打开和关闭。

### 回滚

- 如果单页菜单化布局引入严重可用性问题，可以先回退到当前的“主舞台 + 辅助面板”布局，再单独调整菜单分组。
