# ADR-0001-20260325 Meeting Architecture

- Status: accepted
- Date: 2026-03-25

## 背景与问题定义

`Meeting` 需要在 PC 浏览器和移动 H5 中提供多人视频会议、白板、共享屏幕、录屏/录音、文字聊天、就位确认和会议纪要能力。系统要求：

- 媒体传输严格保留 P2P 优先
- 不支持直连的 peer 才退化为 TURN 中继
- 会议结束后删除会议过程数据
- 服务器仅保留基础审计数据
- 录屏/录音不上传服务器，优先本地缓存与本地下载
- 项目最终使用 MIT 许可开源

## 目标

- 在浏览器/H5 能力边界内实现 P2P 优先的多人会议系统
- 将会议运行态和持久态严格分离
- 服务端只持久化审计与用户偏好，不保留会议正文数据
- 为后续真实 WebRTC、WSS 信令、白板同步和登录体系预留稳定边界

## 非目标

- 第一阶段不做 SFU
- 第一阶段不做服务端集中录制
- 第一阶段不承诺 10 人同时 2K 摄像头
- 第一阶段不实现微信/邮箱登录完整链路

## 决策结论

1. 媒体面使用 WebRTC，全量采用 Mesh 拓扑，默认 P2P 建链，失败时使用 TURN relay。
2. 控制面统一走 WSS，承载信令、权限审批、白板事件、聊天、就位确认和审计上报。
3. 会议运行态仅保留在服务端内存，会议结束后立即清理。
4. 服务器持久化范围严格限制为基础审计数据与已注册用户默认媒体偏好。
5. 录屏/录音仅在客户端本地缓存，用户选择下载时落地到本地路径，放弃时清理缓存。
6. `participant` 默认只有聊天权限，麦克风、摄像头、白板、共享屏幕、录制、就位确认均需主持人授权。

## 方案概述与核心决策

### 总体拓扑

- 浏览器到浏览器：WebRTC P2P
- 浏览器到 coturn：ICE/STUN/TURN
- 浏览器到后端：WSS/HTTPS
- 后端到 SQLite：仅审计与用户偏好

### 数据生命周期

- 会议、参与者、权限、白板、聊天、会议纪要：只保留临时态
- 主持人主动结束会议，或所有人退出后自动结束会议
- 结束时删除临时数据
- 审计事件独立持久化

### 媒体策略

- 默认摄像头低到中码率
- 屏幕共享优先清晰度
- 2K 只作为单路聚焦或屏幕共享目标，不作为全房间默认策略

## 涉及模块

- `meeting-api`: REST API 和后续 WSS 网关
- `meeting`: 房间、角色、权限、生命周期管理
- `audit`: 审计事件入库
- `identity`: 注册用户偏好和登录身份
- `web`: 前端控制台、后续 WebRTC 管理与 UI

## 数据结构与存储影响

- SQLite 持久表：`users`、`user_preferences`、`audit_events`
- 运行时内存态：`meetings`、`participants`、`temporary_minutes`

## 兼容性、迁移方案与风险

- 风险 1：纯 Mesh 在 10 人房间的上行和 CPU 压力较高，需要严格的分辨率/码率治理。
- 风险 2：移动 H5 的共享屏幕支持弱于 PC 浏览器。
- 风险 3：服务端不保留会议正文数据意味着故障恢复能力弱，需要前端和服务端尽量减少状态漂移。

## 验证与回滚思路

- 单元测试验证会议生命周期和权限规则
- 集成测试验证创建、加入、授权、结束、清理、审计持久化
- 若 P2P-first 无法满足质量要求，再补新的 ADR 讨论混合拓扑或 SFU

## 备选方案及放弃原因

### LiveKit / SFU 优先

- 优点：稳定、录制友好、多方房间扩展更容易
- 放弃原因：与“严格保留 P2P 优先”的产品预期不符

### 服务端集中录制

- 优点：录制统一、后处理更容易
- 放弃原因：与“本地录制、不上传服务器”的要求冲突
